## 2026-06-16

- **`exec npm start` em container = npm vira PID1 e ENGOLE o SIGTERM → porta presa → crash loop EADDRINUSE.** Incidente 16/06: o `bk-vexo` entrou em crash loop. Cadeia: healthcheck (`/health`, que pingava o banco, `timeout 12s`) falhava com DB lento → Easypanel reinicia → SIGTERM ia pro **npm (PID1)**, que não repassa o sinal ao node filho → node antigo seguia vivo **segurando :3001** até SIGKILL → o novo processo `app.listen(3001)` batia **EADDRINUSE**, e como o `app.listen` **não tinha `.on('error')`**, o evento `'error'` não tratado fazia `throw` e matava o processo → loop. **Correções (defesa em camadas):** (a) `exec node src/server.js` no start.sh (mantendo `node scripts/conditional-migrate.mjs` antes, = o `npm prestart`) para o node ser PID1 e receber o sinal direto; (b) `server.on('error')` com retry curto e limitado no EADDRINUSE (`LISTEN_RETRY_MAX=3` × `LISTEN_RETRY_DELAY_MS=3000` = 9s) e `exit(1)` controlado se persistir; (c) SIGTERM/SIGINT → `gracefulShutdown` que faz `server.close()` (libera a porta) → `pool.end()` → `exit(0)`, com timeout de força (`SHUTDOWN_FORCE_MS=10000`, `unref()`) pra nunca pendurar; (d) `process.on('uncaughtException'|'unhandledRejection')` logando stack — antes o `throw` do evento `'error'` matava sem rastro. **Lições:** em container, sempre `exec` o binário final (node), nunca deixar npm/bash entre o PID1 e o processo; todo `app.listen` precisa de handler de `'error'`; SIGTERM tem que fechar o servidor HTTP (liberar a porta), não só o pool.

- **Healthcheck deve testar LIVENESS (processo vivo), não READINESS (banco OK).** Mesmo incidente: o `HEALTHCHECK` do Docker batia em `/health`, que faz `postgresHealthPing`. Com o banco lento (afogamento da Evolution), o ping passava do `timeout` → 3 falhas → container `unhealthy` → restart. Reiniciar a API **não conserta** um banco lento — só gera restart-loop sob carga (e foi o gatilho do loop acima). **Correção (opção B):** rota nova `GET /health/live` (responde 200 instantâneo, **sem tocar no banco**) vira o alvo do `HEALTHCHECK` (Dockerfile + docker-compose.prod), com `start-period` maior (60s) p/ cobrir migrations+boot; `/health` (com DB) fica para readiness/monitoramento. **Lição:** liveness = "o processo está de pé e atendendo?"; readiness = "as dependências estão OK?". Acoplar as duas no healthcheck do orquestrador transforma "dependência lenta" em "mata o container". Trade-off aceito: DB down → container fica `healthy` e requests falham (correto: não adianta reiniciar uma API saudável porque o Postgres caiu).

- **`/instance/fetchInstances` da Evolution pode derrubar o painel quando há muitas mensagens e falta índice de expressão em `"Message"`.** Incidente 15-16/06: abrir a tela de instâncias da Evolution e/ou o `EvolutionAdmin` do CRM martelava `GET /instance/fetchInstances`; a Evolution gerava várias queries `SELECT COUNT(*) FROM "Message" WHERE "instanceId" = $1 AND "key"->>'remoteJid' = $2 AND ("key"->>'fromMe')::boolean = false AND "status" = $3`, saturando I/O/CPU e deixando o EasyPanel em 502/pending. **Correções:** (a) no CRM, a consulta remota da Evolution não roda mais no carregamento do admin; só com `?remote=true`/botão "Buscar Evolution"; (b) backend tem cache curto + dedupe de chamadas concorrentes para N requests virarem 1 chamada real; (c) no banco da Evolution, criar índice `idx_message_instance_remotejid_status_fromme_false` em `"Message" ("instanceId", ("key"->>'remoteJid'), "status") WHERE (("key"->>'fromMe')::boolean = false)`. Lição: qualquer endpoint "listar instâncias" que agrega unread/status por mensagem precisa ser tratado como operação cara, não como polling/refetch automático.

- **`CREATE TABLE IF NOT EXISTS` sobre tabela já existente é no-op SILENCIOSO — schema novo do arquivo nunca aplica.** Padrão de bug confirmado em produção (incidente 16/06). A migration `20260516100000` "recriava" `lead_messages` com colunas `lead_phone/role/content`, mas a tabela já existia (criada por `20260420000009` com `phone/sender_type/direction/message_text`). O `IF NOT EXISTS` fez a migration passar sem erro **e sem aplicar nada** → o código escrito contra o schema do arquivo (`chatbot-ai-engine.js` inserindo `lead_phone`) quebrou em runtime (`column "lead_phone" does not exist`), e ficou meses sem persistir o histórico do chatbot, falhando em silêncio (fire-and-forget com só `console.warn`). **Lições:** (a) para MUDAR o schema de uma tabela existente use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` / `DROP CONSTRAINT`, nunca um segundo `CREATE TABLE IF NOT EXISTS` com shape diferente — este é texto morto enganoso; (b) o banco é a verdade, o arquivo de migration é só expectativa: confirmar com `\d tabela` antes de escrever código contra um schema; (c) writer fire-and-forget que só dá `console.warn` mascara o erro — vale logar com destaque ou ter um teste de smoke do INSERT.

- **`status_conversa` é uma MÁQUINA DE ESTADOS DE ATENDIMENTO, não um buffer de evento de disparo.** Bug 2 do incidente: `onLeadDispatched` gravava `status_conversa: "campanha_enviada"` em `lead_import_items`, valor fora da CHECK (`aguardando_usuario|em_atendimento|finalizado`) → violava a constraint a cada disparo. A coluna modela o estado da CONVERSA (esperando o lead / em atendimento / finalizado); "campanha enviada" é um EVENTO de disparo, que já tem seus próprios registros (`followup_status`, `ultima_interacao_bot`, `campaign_dispatch_runs`). **Lição:** antes de inventar um valor novo num campo com CHECK, perguntar "este campo é uma máquina de estados de QUÊ?" e checar se o conceito não pertence a outra coluna. Aqui `"campanha_enviada"` era write-only (nenhum leitor) — sintoma claro de estado órfão; o pós-disparo é `"aguardando_usuario"` (esperando resposta), e o sinal de fila de follow-up é a coluna dedicada `followup_status: "pending"`. Ampliar a CHECK teria adicionado um estado que ninguém lê.

- **`npx tsc --noEmit` no root `tsconfig.json` NÃO checa os arquivos de `src/` neste projeto.** O root tem `"files": []` + só `references` para `tsconfig.app.json`/`tsconfig.node.json`. Isso deu falsa confiança em 3 fatias seguidas e deixou passar um `Trash2` não importado (import órfão cortado por engano) → `ReferenceError` em produção. **Comando correto de validação: `npx tsc --noEmit -p tsconfig.app.json`** (pega o caso como `TS2304: Cannot find name`). Alternativa: `npx tsc -b`.
- **`vite build` (esbuild) NÃO pega identificador não importado.** O esbuild transpila arquivo-a-arquivo e trata nome desconhecido como global → builda com sucesso e só estoura em runtime (`ReferenceError`). Premissa "o build pega referência não resolvida que o type-check deixa passar" está **invertida** para esta classe de bug: quem pega é o type-check contra o tsconfig certo, não o build. O build serve para erros de sintaxe/resolução de módulo, não para identificadores indefinidos.
- **Critério prático de validação:** `tsc -p tsconfig.app.json` hoje acusa erros pré-existentes em arquivos não relacionados (arquivos lixo `" 2".tsx`, `ChatbotConfig.tsx`, e a linha do `handleSubmit` em `Tenants.tsx`). Logo o gate é "nenhum erro NOVO nos arquivos que toquei", não "saída totalmente limpa".
- **Extração de componente exige varrer referências órfãs em AMBAS as direções.** Ao extrair `EvolutionChipsPanel` do `Tenants.tsx`, sobrou `handleCreateEvolutionInstance` que ainda referenciava `getEvolutionDraft`/`saveEvolutionInstance`/`setEvolutionDrafts` (já movidos). A função não era chamada, mas era +3 `ReferenceError` latentes. Lição: depois de extrair, rodar `tsc -p tsconfig.app.json` e procurar `TS2304` no arquivo de origem.

- **Elegibilidade idempotente = claim ANTES do envio, não marcação depois.** No anti-reenvio do disparo (Defeito A), o padrão correto é `INSERT ... ON CONFLICT (dispatch_id, lead_id) DO NOTHING` com status `'claimed'` ANTES de chamar a Evolution; se a linha não foi inserida (conflito), o lead já foi tocado → pula. Marcar só DEPOIS do envio não sobrevive a crash no meio do loop (loop Express, fire-and-forget): na retomada o lead reaparece e duplica. Detalhes: (a) o claim vem DEPOIS de reservar o chip e ANTES do `postEvolutionPayload` — se pular por conflito, devolver a cota reservada (`activeChip.release()`), senão vaza cota; (b) escopo é POR DISPARO (`dispatch_id`), nunca histórico global do lead — senão quebra follow-up legítimo em campanhas futuras; (c) `failed` conta como tocado (sai do reprocesso) e fica exportável para tratamento manual — reprocesso automático de falha mascara chip banido/desconectado.
- **Reutilizar tabela equivalente em vez de criar nova.** `campaign_dispatch_runs` já existia (log por disparo) — estendida com `lead_id`/`claimed_at`/UNIQUE em vez de criar tabela de controle paralela. `UNIQUE(dispatch_id, lead_id)` com `lead_id` que pode ser NULL em linhas legadas funciona: no Postgres NULLs são distintos, então linhas antigas (lead_id NULL) não colidem e o `ON CONFLICT` cobre só as novas.
- **Validação backend sem DB acessível: harness HTTP local.** Sem acesso ao Postgres/Easypanel daqui, dá para exercitar a orquestração de `dispatchCampaignSequence` de verdade subindo um `http.createServer` que faz de Evolution API e dirigindo o motor com callbacks fake (claim→skip+release, falha→onLeadFailed, sucesso conta 1x). Prova a lógica que escrevi; só o `ON CONFLICT` real do Postgres fica para o gate live. `node --check` pega sintaxe; o load ESM (`await import(...)`) pega referência/import indefinido — equivalente backend da lição do Trash2.

## 2026-06-10

- **`ALTER TABLE ADD COLUMN IF NOT EXISTS` pega lock `ACCESS EXCLUSIVE` no PostgreSQL MESMO quando a coluna já existe.** O no-op (coluna já existe → nada acontece) só é decidido DEPOIS que o lock é adquirido. Comportamento diferente de `CREATE TABLE IF NOT EXISTS` (que é mais leve). Consequência direta: colocar um `ALTER TABLE` dentro de uma função de leitura chamada a cada request enfileira locks `ACCESS EXCLUSIVE` sucessivos na tabela — bloqueando todas as queries concorrentes que precisem dela (inclusive `SELECT`). Sintoma observado: disparo ficava preso em `running`, 0 enviados, indefinidamente, sem exceção nem timeout.

- **Schema migration (`ALTER`/`CREATE`) NUNCA deve rodar no caminho quente.** O padrão `ensure*()`  chamado a cada request (ex: `getLeadClientEvolutionInstances` → `ensureLeadClientEvolutionInstancesTable()` → `ALTER TABLE`) é um antipadrão de migration. A alternativa correta: (a) memoizar com flag de módulo (`let _ensured = false`) para rodar uma vez por processo, ou (b) separar em passo de boot explícito. Regra: se uma função de leitura chama `ensure*`, o `ensure*` deve ser trivialmente barato depois da primeira chamada.

- **Bug da Fatia 3a (2026-06-10):** Dois `ALTER TABLE ADD COLUMN IF NOT EXISTS` inseridos dentro de `ensureLeadClientEvolutionInstancesTable()` (chamada a cada leitura de instâncias no disparo) → locks `ACCESS EXCLUSIVE` enfileirados → `await pgDatabasePool.query(...)` pendurado → sem exceção → disparo nunca saía de `running`, 0 enviados. Diagnóstico feito por análise de código sem acesso aos logs do Easypanel (Claude Code não acessa o painel remoto). Confirmação definitiva via `pg_stat_activity` no banco ou logs do Easypanel (verificar colunas `wait_event_type = 'Lock'`). Fix: mover os `ALTER`s para dentro de um bloco `if (!_evolutionInstancesSchemaEnsured)` — roda na primeira chamada e nunca mais.
